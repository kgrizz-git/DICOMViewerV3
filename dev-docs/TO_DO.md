- [ ] **[P2]** **`annotation_manager` skips overlay bitmap/path rendering when `coordinates` is empty.** In `create_presentation_state_items()`, every annotation goes through `_transform_coordinates()` first and the method `continue`s immediately when that returns an empty list. That means an `OVERLAY` annotation with valid `overlay_data` / `paths` / dimensions but no pre-populated `coordinates` never reaches the later `OVERLAY` branch at all. Current tests had to include a dummy coordinate just to exercise the bitmap-item path. Decide whether overlay annotations should bypass the generic coordinate-presence gate, or whether `_parse_overlay_data()` must always synthesize at least one coordinate.
- [ ] **[P3]** **Overlay bitmap bit-order fallback looks inconsistent with the documented DICOM path.** `_convert_overlay_bitmap_to_graphics()` and `_create_overlay_bitmap_item()` use `numpy.unpackbits(..., bitorder='little')`, matching the comments that DICOM overlay bits are LSB-first. But the no-NumPy fallback inside `_convert_overlay_bitmap_to_graphics()` reads bits with `(byte_val >> (7 - bit_idx)) & 1`, which is MSB-first. If that fallback ever runs, overlay pixels may render mirrored/reversed relative to the main path. Align the fallback extraction logic and add a regression test that forces the non-NumPy branch.

- [ ] **[P3]** **Palette index clamping uses only the Red LUT's length.** `core/dicom_palette.py`'s `apply_palette_luts` clamps all three channels' indices against `len(red_lut) - 1` (kept as-is in the 2026-07-09 first-value fix above, deliberately not fixed here). A Green or Blue LUT shorter than Red then raises `IndexError` at lookup, which the shared `except Exception` swallows — the image **silently** falls back to grayscale (returned PIL image has mode `L`, not `RGB`) with only a `print` to stdout. Verified with a synthetic 2-entry Green LUT (`test_palette_color_short_green_lut_falls_back_to_grayscale`). Clamp per-LUT, or validate that all three lengths agree and surface a real warning.
- [ ] **[P3]** **Flat-image normalization casts raw values to `uint8` instead of zeroing.** `dataset_to_image` has three normalize-to-uint8 sites. The color per-channel path (~lines 469-479) zeroes a channel when `max == min`; the grayscale no-window path (~512-516) and the color whole-array fallback (~482-485) instead cast the un-normalized values straight to `uint8`. A constant-valued image of `1000` therefore renders as `232` (wraparound) on two paths and `0` on the third. Degenerate inputs only, but clearly unintended — pick one behavior. Note the `dataset_to_image` complexity refactor deliberately preserves both (extracting `normalize_to_uint8` and `normalize_channels_to_uint8`), so this is a follow-on decision, not a refactor regression.
- [ ] **[P3]** **Revisit enhanced-CT multi-frame merge bookkeeping only if it becomes visibly slow again.** The P1 post-Continue stall was fixed by making `FrameDatasetWrapper` a metadata view rather than deep-copying every non-pixel element per frame: 26.4 s -> 1.16 s full merge for the 364-frame / 182 MB enhanced CT benchmark. The remaining `series_multiframe_info` rebuild measured 575.7 ms, but first display after UI handoff was 178.1 ms and navigator work was small.
