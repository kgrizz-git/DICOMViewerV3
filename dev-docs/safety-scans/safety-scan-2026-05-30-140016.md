## Findings

### Finding 1: Raw exception text is surfaced to end users in multiple dialogs
- **Severity**: MEDIUM
- **Category**: Error Handling / Data Exposure
- **Location**:
  - `src/gui/dialogs/screenshot_export_dialog.py:561`
  - `src/gui/dialogs/radiation_dose_report_dialog.py:169`
  - `src/gui/dialogs/radiation_dose_report_dialog.py:187`
  - `src/gui/dialogs/structured_report_browser_dialog.py:496`
  - `src/gui/dialogs/structured_report_browser_dialog.py:521`
  - `src/gui/dialogs/study_index_search_dialog.py:568`
- **Evidence**:
  - `QMessageBox.critical(self, "Export failed", str(e))`
  - `QMessageBox.warning(self, "Export", f"Could not write file:\n{e}")`
  - `QMessageBox.critical(self, "Study index", f"Remove failed:\n{e}")`
- **Description**: Several user-facing dialogs pass raw exception text directly into `QMessageBox`. These messages can include absolute filesystem paths, permission details, filenames, and internal implementation text from Python/OS exceptions.
- **Attack or Failure Scenario**: A user triggers a failed export, delete, or filesystem operation. The raised exception is shown verbatim, disclosing local machine paths or internals to any person viewing the screen or logs/screenshots captured during support workflows.
- **Impact**: Moderate information disclosure risk and inconsistent privacy posture. The repo already has sanitization utilities and uses them in safer patterns elsewhere, so these unsanitized surfaces are avoidable regressions.
- **Proof of Concept**: Force an export failure by choosing a protected path or disconnected destination; the dialog will display the raw exception string.
- **Recommended Remediation**: Replace raw exception display with a generic user message plus sanitized details, for example `sanitize_message(str(e), redact_paths=True)`. The safer pattern already exists in `src/gui/dialogs/study_index_search_dialog.py:466-470`.
- **Status**: Resolved on 2026-05-30. The affected dialogs now sanitize exception text with `sanitize_message(..., redact_paths=True)`, and focused regression tests cover the updated failure paths.

### Finding 2: `detect-secrets` in CI is advisory only and does not enforce failures
- **Severity**: LOW
- **Category**: Secret Scanning / CI Security
- **Location**: `.github/workflows/security-checks.yml:123-128`
- **Evidence**:
  - `continue-on-error: true`
  - `detect-secrets scan --all-files --force-use-all-plugins > baseline.json || true`
  - `detect-secrets audit baseline.json 2>/dev/null || true`
- **Description**: The `detect-secrets` step is configured so that install or scan failures do not fail the workflow, and the `audit` subcommand is not a meaningful non-interactive CI gate. In practice this provides extra signal, but not an enforceable security control.
- **Attack or Failure Scenario**: A secret pattern missed by TruffleHog verified mode, or a secret flagged only by `detect-secrets`, may not block CI because the step is intentionally non-failing.
- **Impact**: Reduced defense in depth. This is not a direct vulnerability by itself, but it weakens the claimed secret-scanning posture.
- **Proof of Concept**: The workflow logic explicitly swallows failures and never asserts on findings from the `detect-secrets` step.
- **Recommended Remediation**: Either convert `detect-secrets` to a deterministic non-interactive gate with a maintained baseline and explicit failure policy, or document it as informational-only and rely on a single enforced secret scanner.
- **Status**: Resolved on 2026-05-30. The workflow now clearly labels `detect-secrets` as informational-only, removes the misleading `|| true` scan/audit pattern, and keeps TruffleHog as the blocking CI secret scanner.

### Finding 3: Diagnostic scripts still print raw tracebacks
- **Severity**: LOW
- **Category**: Logging / Diagnostic Safety
- **Location**:
  - `scripts/diagnose_multiframe.py:138-140`
  - `scripts/test_multiframe_fix.py:168-170`
  - `scripts/test_multiframe_fix.py:199-201`
- **Evidence**: `traceback.print_exc()` is called directly in these scripts.
- **Description**: Repo diagnostic scripts still emit raw tracebacks to console. These are not main application paths, but they can still expose absolute paths and dataset-specific context during developer or support use.
- **Attack or Failure Scenario**: A developer or support operator runs a diagnostic script against a real dataset and shares console output containing local paths or sensitive context.
- **Impact**: Low but real privacy/logging hygiene gap, especially in shared troubleshooting transcripts.
- **Proof of Concept**: Trigger any exception path in the listed scripts; the full traceback is printed unsanitized.
- **Recommended Remediation**: For scripts expected to be shared or used with real data, prefer sanitized exception formatting or add clear warnings that raw local traceback data may be emitted.
- **Status**: Resolved on 2026-05-30. The affected scripts now default to sanitized traceback output and require `DICOM_DEBUG_TRACEBACKS=1` for raw traceback printing.
