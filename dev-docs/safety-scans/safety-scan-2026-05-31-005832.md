## Findings

### Finding 1: Unsanitized exception text in 3 export dialog error messages

- **Severity**: LOW
- **Category**: Error Handling / Data Exposure (CWE-209)
- **CWE**: CWE-209
- **Location**: `src/gui/dialogs/export_roi_statistics_dialog.py` line 394, `src/gui/dialogs/export_dialog.py` line 663, `src/gui/dialogs/tag_export_dialog.py` line 796
- **Evidence**:
  ```python
  # export_roi_statistics_dialog.py:394
  f"Could not write file:\n{str(e)}\n\nYou can change the path and try again."

  # export_dialog.py:663
  f"Failed to export files:\n{str(e)}"

  # tag_export_dialog.py:796
  f"Failed to export tags:\n{str(e)}"
  ```
- **Description**: Three export dialogs display raw `str(e)` exception text in user-facing `QMessageBox.critical()` calls without using `sanitize_message()`. While other dialogs (screenshot export, study index search, radiation dose report, structured report browser) properly use `sanitize_message(str(e), redact_paths=True)`, these three do not.
- **Attack or Failure Scenario**: If an export fails due to a filesystem error, the raw exception may reveal full file paths, internal directory structure, or OS-level error details to the user. In a shared-screen or screenshot scenario, this could expose directory layout.
- **Impact**: Low — desktop app where the user already has access to their own filesystem. Risk is limited to incidental path disclosure in error screenshots/recordings.
- **Recommended Remediation**: Replace `str(e)` with `sanitize_message(str(e), redact_paths=True)` in all three locations, consistent with the pattern already used elsewhere.
- **Regression Test**: Add test cases for export error paths verifying sanitized output.
- **Status**: Resolved — `sanitize_message(str(e), redact_paths=True)` applied to all 3 locations

### Finding 2: starlette 1.0.0 transitive dependency vulnerability (PYSEC-2026-161)

- **Severity**: MEDIUM
- **Category**: Dependency Vulnerabilities
- **CWE**: N/A (CVE-level dependency issue)
- **Location**: `.venv/` — transitive dependency via `mcp` → `sse-starlette` → `starlette`
- **Evidence**:
  ```
  pip-audit output:
  Name      Version ID             Fix Versions
  starlette 1.0.0   PYSEC-2026-161 1.0.1
  ```
- **Description**: `starlette` 1.0.0 has a known vulnerability (PYSEC-2026-161), fixed in 1.0.1. Starlette is a Python ASGI web framework — it is **not used by the DICOM Viewer application at all**. It is a transitive dependency brought in by the `mcp` package (via `mcp` → `sse-starlette` → `starlette`). The `mcp` package is not listed in `requirements.txt` and is not part of the application — it was installed separately into the local `.venv` for AI developer tooling (Claude Code MCP server support).
- **Attack or Failure Scenario**: Starlette vulnerabilities affect web server deployments. This application is a desktop DICOM viewer that does not run a web server, does not import starlette, and does not serve HTTP requests. The **effective attack surface is zero** for the application itself. The only scenario where this matters is if an MCP server is run locally from this venv, which is a developer workflow concern, not an application concern.
- **Impact**: Negligible for the application. Low for developer environments that run MCP servers from this venv.
- **Recommended Remediation**: Update `mcp` package to a version that pulls `starlette >= 1.0.1`, or pin `starlette>=1.0.1` in requirements. Run: `pip install --upgrade mcp starlette`
- **Regression Test**: N/A — dependency version check
- **Status**: Open (carried forward from previous scan 2026-05-30)

### Finding 3: CI actions pinned to version tags instead of commit SHAs

- **Severity**: LOW
- **Category**: CI/CD Pipeline Security (Supply Chain)
- **Location**: `.github/workflows/security-checks.yml`, `semgrep.yml`, `grype.yml`
- **Evidence**: Actions use version tags like `@v6`, `@v7`, `@v9`, `@v4` instead of full commit SHAs
- **Description**: GitHub Actions best practice recommends pinning third-party actions to full commit SHAs rather than mutable version tags. Tags can be moved to point to different code, creating a supply-chain risk. `trufflesecurity/trufflehog@v3.95.2` is better but still a tag.
- **Impact**: Low — risk requires compromise of action maintainer's GitHub account or repository. All actions used are from well-known, trusted publishers (GitHub official, Anchore, TruffleSecurity).
- **Recommended Remediation**: Pin actions to full commit SHAs with version comments. Example:
  ```yaml
  - uses: actions/checkout@<full-sha>  # v6
  ```
- **Status**: Open (defense-in-depth improvement)

### Finding 4: `core.hooksPath` not configured; hooks rely on manual installation

- **Severity**: LOW
- **Category**: Development Workflow Security
- **Location**: Git configuration
- **Evidence**: `git config core.hooksPath` returns "not set"; hooks exist in both `.githooks/` and `.git/hooks/`
- **Description**: The project maintains hooks in `.githooks/` (version-controlled) but does not set `core.hooksPath` to point there. Hooks appear to be manually copied or symlinked to `.git/hooks/`. New clones or contributors must manually install hooks.
- **Impact**: Low — new contributors could skip pre-commit checks if hooks are not installed.
- **Recommended Remediation**: Add `git config core.hooksPath .githooks` to the project setup instructions or a setup script. Alternatively, document the manual install step prominently.
- **Status**: Resolved — `setup-hooks.sh` and `install-local-git-hooks.ps1` now set `core.hooksPath` to `.githooks/`; `DEVELOPER_SETUP.md` updated; `core.hooksPath` set locally
